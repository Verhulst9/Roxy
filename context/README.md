## 模块概述

### 职责定位
Context 模块是 Nakari 的**记忆管理中枢**，负责管理和优化 LLM 的 Context Window。该模块确保在有限的 Token 预算内，为 LLM 提供最相关的上下文信息，同时保持对话历史的连续性和关键信息的完整性。

### 设计目标
1. **智能检索**：根据当前对话上下文，从记忆系统中检索最相关的原子
2. **动态压缩**：当 Context Window 接近满时，自动压缩历史对话
3. **优先级管理**：确保关键信息（如用户偏好、长期记忆）不会被丢弃
4. **连续性保持**：维护对话的上下文连贯性，避免信息断层
5. **性能优化**：最小化检索和压缩的计算开销

### 技术选型

| 技术 | 版本 | 选型理由 |
|------|------|----------|
| LangChain | 0.1+ | 成熟的 LLM 应用框架，提供丰富的上下文管理工具 |
| LangGraph | 0.2+ | 状态图编排，支持复杂的上下文流转 |
| 向量数据库 | 待定 | 用于语义检索，支持高效的相关性匹配 |

## 设计理念

### 上下文窗口的生命周期

```
┌─────────────────────────────────────────────────────────────┐
│                    Context Window 生命周期                     │
└───────────────────────┬─────────────────────────────────────┘
                        │
         ┌──────────────┼──────────────┐
         ▼              ▼              ▼
    ┌─────────┐   ┌─────────┐   ┌─────────┐
    │ 检索阶段 │   │ 组装阶段 │   │ 压缩阶段 │
    │ Retrieval│   │ Assembly│   │Compression│
    └────┬────┘   └────┬────┘   └────┬────┘
         │             │             │
         ▼             ▼             ▼
    从记忆系统中   组装最终上下文  压缩历史对话
    检索相关原子   发送给 LLM    释放 Token 空间
```

### 上下文构成

**Context Window = 系统提示 + 对话历史 + 检索记忆 + 用户输入**

| 组成部分 | 内容 | Token 占比 | 优先级 |
|---------|------|-----------|--------|
| 系统提示 | Nakari 的人格定义、行为准则 | 10-20% | 不可丢弃 |
| 对话历史 | 近期的对话记录 | 30-40% | 可压缩 |
| 检索记忆 | 从记忆系统中检索的原子 | 40-50% | 动态调整 |
| 用户输入 | 当前用户的输入 | 5-10% | 不可丢弃 |

## 核心组件

### 1. Context Retriever（上下文检索器）

**职责**：
- 分析用户输入，识别关键实体和话题
- 生成查询向量，从记忆系统中检索相关原子
- 进行图遍历扩展，获取相关的上下文网络

**接口设计**：
```python
def retrieve_context(
    user_input: str,
    conversation_history: List[Dict[str, str]],
    max_tokens: int = 2000
) -> RetrievedContext:
    """
    检索相关上下文

    参数:
        user_input: 用户输入
        conversation_history: 对话历史
        max_tokens: 最大 Token 预算

    返回:
        RetrievedContext {
            atoms: List[Atom],           # 检索到的原子
            relevance_scores: List[float], # 相关性分数
            total_tokens: int,            # 总 Token 数
        }
    """
```

### 2. Context Assembler（上下文组装器）

**职责**：
- 组装完整的上下文（系统提示 + 对话历史 + 检索记忆 + 用户输入）
- 计算 Token 使用量
- 按优先级排序上下文元素

**接口设计**：
```python
def assemble_context(
    system_prompt: str,
    conversation_history: List[Dict[str, str]],
    retrieved_atoms: List[Atom],
    user_input: str,
    max_tokens: int = 4096
) -> AssembledContext:
    """
    组装上下文

    参数:
        system_prompt: 系统提示
        conversation_history: 对话历史
        retrieved_atoms: 检索到的原子
        user_input: 用户输入
        max_tokens: 最大 Token 预算

    返回:
        AssembledContext {
            context: str,           # 组装后的上下文
            token_count: int,       # Token 计数
            included_atoms: List[str], # 包含的原子 ID
            summary_needed: bool,   # 是否需要压缩
        }
    """
```

### 3. Context Compressor（上下文压缩器）

**职责**：
- 使用摘要方式压缩历史对话
- 保留关键信息的语义
- 自动维护对话历史的摘要版本

**接口设计**：
```python
def compress_history(
    conversation_history: List[Dict[str, str]],
    target_tokens: int
) -> CompressedHistory:
    """
    压缩历史对话

    参数:
        conversation_history: 对话历史
        target_tokens: 目标 Token 数量

    返回:
        CompressedHistory {
            summary: str,              # 压缩后的摘要
            original_tokens: int,      # 原始 Token 数
            compressed_tokens: int,    # 压缩后 Token 数
            compression_ratio: float,  # 压缩比例
        }
    """
```

## 上下文压缩策略

### 压缩触发条件

| 触发条件 | 阈值 | 处理策略 |
|---------|------|---------|
| Token 使用率 | > 85% | 压缩对话历史 |
| 对话轮数 | > 20 轮 | 压缩早期对话 |
| 摘要陈旧度 | > 5 轮未更新 | 重新生成摘要 |

### 压缩算法

**分级压缩策略**：

1. **第一级：对话摘要**
   - 将早期对话轮次（第 1-10 轮）压缩为摘要
   - 保留关键实体、用户偏好、重要事件

2. **第二级：原子压缩**
   - 对检索到的原子进行二次过滤
   - 仅保留 Top-K 高相关性原子

3. **第三级：系统提示优化**
   - 压缩冗长的系统提示
   - 移除重复的人格定义

### 摘要生成

**摘要模板**：
```
[对话摘要]
- 用户主要话题：[话题列表]
- 用户偏好：[偏好列表]
- 重要事件：[事件列表]
- 持续讨论的问题：[问题列表]
```

**LLM 提示示例**：
```
请将以下对话压缩为简明的摘要，保留关键信息：
- 用户的主要兴趣和偏好
- 重要的讨论话题
- 需要记住的上下文信息
- 用户提到的重要实体和事件

对话内容：
[对话历史]

输出格式：
- 主要话题：...
- 用户偏好：...
- 重要事件：...
```

## 内存管理

### Token 预算分配

| 上下文部分 | 最低 Token | 推荐 Token | 最大 Token |
|-----------|-----------|-----------|-----------|
| 系统提示 | 200 | 500 | 800 |
| 对话历史 | 500 | 1500 | 2500 |
| 检索记忆 | 800 | 2000 | 3000 |
| 用户输入 | 100 | 300 | 500 |
| **总计** | **1600** | **4300** | **6800** |

### 动态调整策略

**场景 1：用户输入较长**
- 降低对话历史的 Token 分配
- 优先保留最新的对话轮次

**场景 2：需要检索大量记忆**
- 压缩对话历史
- 提高检索记忆的 Token 分配

**场景 3：多轮对话**
- 渐进式压缩早期对话
- 保留最近 10-15 轮的完整对话

## 性能优化

### 检索优化

**向量检索优化**：
- 使用近似最近邻算法（ANN）加速检索
- 建立索引缓存，减少重复检索
- 批量检索相关原子

**图遍历优化**：
- 限制遍历深度（建议 2-3 层）
- 使用剪枝策略，避免无限扩展
- 缓存常用子图

### 压缩优化

**增量压缩**：
- 只压缩新增的对话轮次
- 复用已有的摘要

**异步压缩**：
- 将压缩任务放入后台队列
- 避免阻塞用户交互

**多级缓存**：


## 最佳实践

### 1. 检索策略

**混合检索**：
- 结合语义检索和关键词检索
- 提高检索准确性和覆盖面

**相关阈值**：
- 设置最低相关性分数（如 0.7）
- 低于阈值的原子不纳入上下文

### 2. 压缩策略

**保持语义完整性**：
- 压缩后应保留对话的核心语义
- 避免丢失用户偏好和重要信息

**定期更新摘要**：
- 每隔 5-10 轮对话，重新生成摘要
- 确保摘要反映最新的对话内容

### 3. 上下文监控

**关键指标**：
- Token 使用率
- 检索准确率（用户满意度）
- 压缩比例
- 上下文完整性（信息丢失率）

**告警阈值**：
- Token 使用率 > 90%
- 压缩比例 > 70%
- 检索准确率 < 80%

## 总结

Context 模块是 Nakari 的记忆管理中枢，通过 LangChain/LangGraph 实现了：

1. **智能检索**：基于语义的上下文检索，确保相关性
2. **动态压缩**：自动压缩历史对话，保持关键信息
3. **优先级管理**：合理分配 Token 预算，确保关键信息不丢失
4. **性能优化**：多级缓存、异步压缩，最小化计算开销
5. **可观测性**：完整的监控指标，支持实时调整策略

该模块的设计充分考虑了 LLM Context Window 的限制，通过智能的检索和压缩策略，在有限的 Token 预算内，为 LLM 提供最相关、最完整的上下文信息。
